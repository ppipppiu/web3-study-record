## 3. 本周完成的内容

### 3.1 明确最小原型

我将本周原型定义为一个链上签到合约。

用户的核心操作是：

1. 连接钱包；
2. 调用 `checkIn()` 完成签到；
3. 调用查询函数查看今天是否已签到；
4. 查看累计签到次数和连续签到天数。

本周只实现智能合约和链上验证，暂时没有开发完整前端页面。

### 3.2 编写签到合约

我完成了 `CheckIn.sol`，合约使用以下核心结构：

```solidity
struct CheckInInfo {
    uint256 totalCheckIns;
    uint256 currentStreak;
    uint256 lastCheckInDay;
}
```

通过以下映射保存每个钱包地址的签到信息：

```solidity
mapping(address => CheckInInfo) private checkInRecords;
```

主要函数包括：

* `getCurrentDay()`：将区块时间换算为当前天数；
* `hasCheckedInToday(address)`：查询指定地址今天是否已经签到；
* `checkIn()`：完成签到并更新累计次数和连续天数。

签到成功后会触发 `CheckedIn` 事件。

### 3.3 Remix 编译和部署

我将合约放入 Remix，并使用 Solidity 0.8.x 编译器完成编译。

编译成功后，我先在 Remix VM 中部署合约并进行基础测试，确认合约可以正常运行，再部署到 Monad Testnet。

### 3.4 第一次签到测试

部署后，我使用第一个测试账户调用 `checkIn()`。

交易执行成功，说明用户在当天第一次签到时，可以正常更新签到数据。

随后调用：

```solidity
hasCheckedInToday(address)
```

查询结果返回 `true`，说明签到状态已经成功保存。

### 3.5 重复签到失败测试

我在同一天使用同一个账户再次调用 `checkIn()`。

交易按预期回滚，并返回：

```text
You have already checked in today
```

这说明合约中的重复签到限制生效，同一个钱包地址每天只能签到一次。

### 3.6 多账户独立性测试

我切换到另一个 Remix 测试账户。

新账户在未签到时可以正常调用 `checkIn()`，说明不同钱包地址拥有独立的签到记录。

这一测试也验证了：

```solidity
mapping(address => CheckInInfo)
```

能够通过 `msg.sender` 区分不同用户。

### 3.7 Monad Testnet 部署

完成 Remix VM 测试后，我将合约部署到 Monad Testnet。

部署完成后，我保存了：

* 合约部署交易 Hash；
* `checkIn()` 签到交易 Hash；
* 合约地址；
* 区块浏览器查询记录。

我还通过链上调用测试了：

* `checkIn()`；
* `hasCheckedInToday(address)`；
* 当天重复签到失败。

## 5. 遇到的错误和解决过程

### 5.1 不清楚如何判断编译成功

刚开始使用 Remix 时，我不知道在哪里确认合约是否编译成功。

后来通过 Solidity Compiler 页面中的成功提示，以及 Deploy & Run Transactions 页面中可以选择合约，确认编译已经完成。

### 5.2 不清楚查询函数和交易函数的区别

开始时，我对按钮颜色和函数调用方式不熟悉。

后来理解到：

* 查询函数通常不会修改链上数据，不需要发送交易；
* `checkIn()` 会修改状态，因此需要钱包确认并消耗测试网 Gas；
* 查询函数返回的是读取结果；
* 写入函数需要查看交易状态和交易 Hash。

### 5.3 两个账户的查询结果看起来相同

在切换账户测试时，我曾发现两个查询结果相同，因此不确定账户数据是否真正独立。

后来重新按照正确顺序测试：

1. 账户 A 完成签到；
2. 查询账户 A，结果为已签到；
3. 查询未签到的账户 B；
4. 切换到账户 B 完成签到；
5. 再分别查询两个账户。

通过明确传入不同地址，最终确认不同账户的数据是独立保存的。

### 5.4 当天重复签到交易失败

第二次调用 `checkIn()` 时，交易被回滚。

这不是程序错误，而是预期结果。合约通过 `require` 主动阻止用户在同一天重复签到，因此这次失败反而证明了限制逻辑有效。

在拆解任务后，我调整了判断：
