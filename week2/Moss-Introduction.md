# 我对 Moss 的理解：让 AI Agent 更安全地与 Monad 交互

## 前言

最近在学习 Web3 开发的过程中，我接触到了 Moss 这个开源项目。

一开始看到“AI Agent × Web3”时，我的理解比较模糊。我原本以为，它可能只是让 AI 帮助用户调用智能合约，或者自动完成一些链上操作。

但在阅读 Moss 的 README、项目目录和部分代码后，我发现 Moss 真正想解决的问题，不只是“让 AI 能够操作区块链”，而是：

> 如何让 AI Agent 在不直接处理复杂合约细节的情况下，更安全、更标准地完成链上交互。

下面是我目前对 Moss 的理解，以及这次阅读和探索项目的过程。

---

## 一、Moss 是什么

Moss 是一个面向 Monad 生态的开源项目。

它将原本比较复杂的 DApp 和智能合约操作，封装成 AI Agent 可以理解和调用的统一能力。

正常情况下，如果要完成一次链上交互，开发者可能需要处理很多内容，例如：

- 合约地址；
- ABI；
- 函数名称；
- 参数格式；
- calldata；
- Token 的 decimals；
- 授权操作；
- 多笔交易之间的执行顺序；
- 交易执行后会发生哪些资产变化。

这些内容对普通用户比较复杂，对 AI Agent 来说也容易出错。

Moss 的作用，就是在 AI Agent 和区块链协议之间增加一层标准化的执行系统。

AI Agent 不需要自己拼接底层交易数据，而是通过 Moss 提供的能力，描述自己想完成什么操作，再由 Moss 负责构建对应的未签名交易。

需要特别注意的是：

**Moss 本身不会替用户签名，也不会直接发送交易。**

私钥仍然保存在用户的钱包中，是否确认交易，最终仍然由用户决定。

---

## 二、Moss 主要解决了什么问题

我认为 Moss 主要解决了三个问题。

### 1. 降低 AI Agent 调用链上协议的难度

不同协议的合约结构、函数名称和参数格式都可能不同。

如果让 AI Agent 直接处理 ABI、合约地址和 calldata，不仅开发难度高，也很容易因为参数错误导致交易失败。

Moss 将这些复杂内容封装成统一的能力。

例如，Agent 只需要表达：

- 查询账户余额；
- 转账某种 Token；
- 将 MON 包装为 WMON；
- 完成一次 Swap；
- 查询某个 NFT 的持有者。

至于具体调用哪个合约、使用什么参数、如何构建交易，则由 Moss 的协议适配层负责。

这相当于把底层的合约调用，转换成了更容易理解的标准接口。

### 2. 在签名前检查交易结果

链上交易一旦签名并执行，通常无法撤销。

如果 AI Agent 构造出错误交易，可能导致：

- 转错资产；
- 转错地址；
- 授权额度异常；
- 实际支出的 Token 超过预期；
- 调用了错误的合约；
- 交易执行结果和用户意图不一致。

因此，Moss 不只是负责生成交易，还会在签名前对交易进行模拟。

模拟完成后，系统会检查：

- 实际转出了什么资产；
- 实际收到了什么资产；
- 是否产生了授权；
- 资产接收方是谁；
- 实际结果是否超出了原本声明的范围。

如果模拟结果中出现了未声明的资产变化，系统就会产生 warning。

按照 Moss 的设计原则，只要存在 warning，就不应该继续执行交易。

### 3. 让不同协议能够使用统一的接入方式

Web3 中有很多不同类型的协议，例如：

- Token；
- NFT；
- 去中心化交易所；
- 借贷协议；
- 链游；
- 质押协议；
- 链上身份系统。

这些协议的合约接口并不统一。

Moss 通过 Adapter，也就是协议适配器，将不同协议封装成相对统一的调用方式。

这样一来，AI Agent 不需要为每个协议重新学习一套完全不同的调用逻辑，而是可以按照 Moss 的标准流程发现和使用不同能力。
