# Week 2 Role Choice Card

## 1. 选择方向

我在 Week 2 选择的主方向是 **Dev**。

## 2. 选择理由

我选择 Dev 方向，是因为我希望通过实际开发一个可以运行和验证的 Web3 Demo，进一步理解智能合约、钱包交互和链上数据之间的关系。

我目前有计算机和数据科学相关的学习背景，也接触过 Python、SQL、Java 和数据分析项目，但对 Web3 开发仍处于入门阶段。因此，我希望从一个功能简单、范围明确的项目开始，通过编写 Solidity 合约、在 Remix 中测试，并部署到 Monad Testnet，建立完整的链上开发认知。

相比只阅读概念，我更希望通过实际操作理解以下内容：

- 智能合约如何保存不同钱包用户的数据；
- 读取函数和写入函数有什么区别；
- 钱包如何发起链上交易；
- 合约部署和函数调用如何在区块浏览器中验证；
- AI 如何辅助代码生成、解释和测试，但最终结果仍需要由开发者检查。

## 3. 希望服务的问题

本周我选择实现一个最小签到合约，主要服务于需要记录用户连续参与行为的链上场景。

例如：

- 社区每日签到；
- 学习任务打卡；
- 游戏每日登录；
- 活动参与记录；
- 用户连续活跃激励。

传统互联网平台中的签到记录通常由中心化服务器保存，用户需要信任平台不会随意修改数据。链上签到可以将用户的签到次数、连续签到天数和最后签到日期记录在智能合约中，使记录更加公开且可验证。

本周的重点不是实现完整产品，而是先验证签到逻辑是否能够在链上正常运行。

## 4. Week 2 最小产出

本周计划完成一个最小可运行的 `CheckIn` 智能合约，主要包括以下功能：

1. 每个钱包地址拥有独立的签到记录；
2. 用户每天只能签到一次；
3. 记录用户累计签到次数；
4. 记录用户当前连续签到天数；
5. 记录用户最后一次签到日期；
6. 支持查询某个地址今天是否已经签到；
7. 签到成功后触发链上事件。

验收方式包括：

- Solidity 合约能够在 Remix 中编译成功；
- 合约能够部署并正常调用；
- 第一次签到成功；
- 同一天重复签到会失败；
- 不同钱包地址的签到数据互不影响；
- 查询函数能够返回正确结果；
- 合约部署和签到交易能够在 Monad Testnet 区块浏览器中查询。

本周的实际产出包括：

- `contracts/CheckIn.sol` 智能合约；
- Remix 编译和测试记录；
- Monad Testnet 合约部署记录；
- `checkIn()` 签到交易记录；
- `hasCheckedInToday(address)` 查询结果；
- 重复签到失败测试；
- GitHub 项目说明和开发记录。

## 6. Week 3 角色方向

Week 3 我希望继续承担 **Dev 角色**，主要负责：

- 智能合约功能设计和开发；

同时，我也希望结合自己的数据分析背景，参与用户行为指标和数据看板设计，例如：

- 总签到人数；
- 总签到次数；

- 连续签到用户数；
- 用户签到频率；

我希望在 Week 3 将智能合约开发与数据分析结合起来，不只完成链上功能，也让链上数据能够被查看和分析。
