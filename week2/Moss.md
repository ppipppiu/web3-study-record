# Moss 开源项目学习笔记

## 一、项目简介

Moss 是一个面向 Monad 生态的开源框架，主要作用是将复杂的 DApp 和链上协议交互，统一封装成 AI Agent 可以理解和调用的能力。

在传统方式中，如果 AI Agent 想完成一次代币交换、Token 转账或 NFT 转移，通常需要自行处理：

* 合约地址
* ABI
* calldata 编码
* Token decimals 换算
* Approve 授权
* 多笔交易的执行顺序
* 滑点、退款和资产流向

这些操作比较复杂。如果参数、合约地址或交易顺序出现错误，就可能导致交易失败，甚至造成资产损失。

Moss 将这些底层操作封装在统一的能力层中。AI Agent 只需要传入相对容易理解的参数，Moss 就可以生成已经组装好的**未签名交易**，并在用户签名前进行模拟和风险检查。

Moss 的核心定位可以概括为：

> 为 AI Agent 提供统一、可发现、可验证的 Monad 链上交互能力。

需要注意的是，Moss 不会保管用户私钥，也不会直接签名或发送交易。最终是否签名，仍然由用户通过钱包决定。

---

## 二、Moss 希望解决的核心问题

### 1. AI Agent 难以安全地直接操作区块链

AI Agent 可以理解用户的自然语言，例如：

> 把 1 MON 换成 USDC。

但自然语言与一笔真实的链上交易之间，仍然存在较大的技术距离。

Agent 如果直接读取 ABI 并自行拼装交易，需要正确处理：

* 应该调用哪个协议和合约
* 应该使用哪个函数
* 参数单位如何转换
* 是否需要先授权
* 授权额度是否合理
* 最终有哪些资产流入和流出
* 多步操作能否按照正确顺序执行

Moss 希望将这些容易出错的底层细节交给标准化框架处理，而不是让每个 Agent 重复编写交易逻辑。

### 2. 不同协议缺少统一的调用方式

不同 DApp 和协议的合约接口、参数结构和交互流程通常都不相同。

例如，同样是“交换代币”，不同去中心化交易所可能会使用不同的：

* 合约地址
* 交易函数
* 参数结构
* 报价方式
* 授权流程

AI Agent 如果要支持多个协议，就需要分别理解并适配每一个协议。

Moss 通过 **Protocol Adapter，协议适配器**，将不同协议封装成统一的能力，使 Agent 可以使用相似的调用方式完成不同协议中的操作。

### 3. 交易在签名前缺少统一验证

仅仅生成一笔“看起来正确”的交易并不够。用户还需要知道：

* 实际会转出多少资产
* 实际会收到多少资产
* 是否出现额外授权
* 收款地址是否正确
* 是否产生了计划之外的资产变化

Moss 会先生成一个 Plan，声明预计发生的资产变化，再通过模拟得到真实的交易 effects，并将二者进行对账。

如果出现未声明的资产变化，Moss 会生成 warning，并提示停止后续签名。

因此，Moss 不只是一个交易生成器，也是一道签名前的安全检查层。

## 三、Moss 的核心工作流程

Moss 的标准调用流程是：

```text
discover → load → action → simulate
```

### 1. discover：发现能力

`discover` 用于查询 Moss 当前支持哪些协议和操作。

例如，Agent 可以查询：

* 哪些协议支持 Swap
* 哪些能力属于 Transfer
* 当前支持哪些 Token 操作
* 当前支持哪些 NFT 操作

这一阶段相当于让 Agent 先了解：

> 当前工具箱中有哪些工具可以使用。

### 2. load：读取调用说明

`load` 用于读取某项能力的详细调用规则，包括：

* 该能力的用途
* 调用时需要哪些参数
* 每个参数分别代表什么
* 可能涉及哪些风险
* 是否会发生资产流出
* 是否涉及授权或价格影响

这一阶段相当于让 Agent 在正式调用前，先阅读工具的使用说明。

### 3. action：生成交易计划

`action` 会根据协议、操作类型、用户账户和参数，生成一个 Plan。

Plan 通常包含：

* `txs`：已经编码好的未签名交易
* `expects`：预计允许发生的资产流入、流出和授权
* `intent`：这组交易希望完成什么目标
* `declaredRisk`：已经声明的风险
* `planHash`：用于检查 Plan 是否被修改的完整性标识

这一阶段只负责构建交易，不负责签名和发送。

### 4. simulate：模拟与验证

`simulate` 会基于当前链上状态模拟执行 Plan，并提取实际发生的效果，例如：

* Token 流入
* Token 流出
* 原生 MON 数量变化
* Approve 授权
* 收款地址
* 多笔交易之间的状态变化

随后，Moss 会将模拟得到的 `effects` 与 Plan 中声明的 `expects` 进行对账。

如果出现计划外变化，系统会产生 warning。

Moss 的安全原则可以概括为：

> 出现 warning 就应当停止，不应该继续交给钱包签名。

## 四、Moss 的核心能力

根据项目文档，Moss 目前主要支持以下类型的链上操作：

### 1. WMON

支持：

* MON 转换为 WMON
* WMON 转换为 MON
* 查询余额

### 2. ERC-20

支持：

* Token 转账
* 查询 Token 余额
* 查询授权额度

### 3. ERC-721

支持：

* NFT 转移
* 查询 NFT 所有者
* 查询 NFT 余额

### 4. Kuru

支持：

* 市价 Swap
* 查询报价
* 查询市场列表

Moss 采用“一个协议对应一个独立 Package”的设计。

开发者可以为新的 Monad 协议编写适配器，从而扩展 Moss 支持的协议和能力。

目前暂不支持的场景包括：

* Permit 或 Typed Data 类型的签名流程
* 跨链桥
* 闪电贷原子组合

## 五、AI Agent 为什么需要 Moss

普通聊天型 AI 可以告诉用户应该怎样操作，但不能保证生成的链上交易一定与用户的真实意图一致。

Moss 为 AI Agent 增加了一个结构化的链上执行层。

完整过程可以理解为：

1. AI Agent 理解用户的自然语言意图
2. Agent 通过 Moss 查找合适的链上能力
3. Moss 构建标准化的未签名交易
4. Moss 模拟交易并检查资产变化
5. Agent 判断模拟结果是否符合用户原始要求
6. 用户在钱包中进行最终确认和签名

这里存在两层检查。

### 第一层：Effects 对账

Moss 负责机械化检查：

> 这笔交易实际上会做什么？

例如检查：

* 实际转出了什么资产
* 实际收到了什么资产
* 是否产生了额外授权
* 是否出现未声明的资产变化

### 第二层：意图对齐

AI Agent 负责检查：

> 交易实际效果是否符合用户最初的要求？

例如用户要求：

> 把 1 MON 换成 USDC，但如果价格太差就不要执行。

即使交易本身可以成功，Agent 仍然需要检查最终收到的 USDC 是否满足用户要求。

这种分工比让 AI Agent 独立完成全部链上操作更安全，也更容易扩展到多个协议。

## 六、可能的应用场景

### 场景一：自然语言 DeFi 助手

用户可以直接输入：

> 帮我把 1 MON 换成 USDC，但如果预计收到的 USDC 太少，就不要执行。

AI Agent 可以通过 Moss 完成以下流程：

1. 使用 `discover` 找到支持 Swap 的协议
2. 使用 `load` 读取 Swap 所需参数和风险
3. 使用 `action` 生成未签名交易 Plan
4. 使用 `simulate` 检查实际资产流向、授权和预计收到数量
5. 确认没有 warning，并且结果符合用户要求
6. 将交易交给用户钱包确认

这种方式可以降低普通用户使用 DeFi 的门槛，同时避免 AI 直接控制用户私钥。

### 场景二：自动化资产管理 Agent

Moss 未来可以应用于定期资产管理，例如：

* 定期将部分 MON 换成稳定币
* 根据预设比例调整 Token 资产配置
* 领取奖励后自动执行 Swap
* 将获得的奖励重新投入某个协议
* 批量处理多个链上操作

例如，一个多步骤操作可能是：

```text
领取奖励 → 将奖励换成 USDC → 将 USDC 存入某个协议
```

Moss 可以先模拟完整流程，检查前一步产生的资产能否被后一步正确使用。

### 场景三：Web3 应用中的智能操作入口

传统 DApp 通常需要为每一种链上操作分别设计：

* 页面
* 表单
* 按钮
* 参数输入框

未来可以在 DApp 中加入自然语言操作入口。

用户只需要说明目标，AI Agent 与 Moss 就可以共同生成可验证的交易。

例如，在链上签到或积分应用中，可以继续扩展：

* 自动领取签到奖励
* 将奖励兑换为指定 Token
* 批量领取多个奖励
* 根据用户设置完成奖励转账
* 在执行前展示资产变化和风险提示

## 七、我的个人理解

我认为 Moss 的重点并不是让 AI 完全代替用户操作钱包，而是在 AI 和区块链之间增加一个标准化、可验证的中间层。

AI Agent 擅长：

* 理解自然语言
* 拆解用户任务
* 选择合适的工具
* 组织多步骤流程

但 AI Agent 并不适合独立承担所有底层链上交易细节。

链上交易涉及真实资产。即使只有一个参数错误，也可能造成不可逆的结果。

Moss 将整个过程进行了职责拆分：

* AI Agent 负责理解用户意图
* Moss 负责构建交易
* Moss 负责模拟交易
* Moss 负责检查实际资产变化
* 用户负责最终确认和签名

我认为 Moss 最有价值的地方主要有三点。

### 1. 降低协议接入成本

Agent 不需要每次都重新读取 ABI、查找合约地址和编写 calldata。

新的协议只需要通过适配器接入 Moss，就可以被不同的 AI Agent 重复使用。

### 2. 提高交易安全性

Moss 会通过 `expects` 和 `effects` 对账，检查实际交易结果是否超出了原计划。

如果出现额外授权、异常资产流出或其他未声明变化，系统可以在签名前发出警告。

### 3. 保留用户控制权

Moss 不保存私钥、不签名，也不直接发送交易。

交易最终仍然需要经过用户的钱包确认，因此用户保留了最终控制权。

不过，Moss 当前仍处于 Alpha 阶段，尚未完成正式安全审计。

另外，模拟结果只能反映模拟时的链上状态。价格、流动性和合约状态可能会在模拟完成后发生变化。

因此，Moss 目前更适合：

* 学习
* 测试
* Demo 开发
* 协议适配验证

不能将“模拟没有 warning”直接理解为交易绝对安全。

---

## 八、总结

Moss 是一个连接 AI Agent 与 Monad 链上协议的基础设施项目。

它希望解决的核心问题是：

> AI Agent 虽然能够理解用户意图，但缺少统一、安全、可验证的链上操作方式。

Moss 通过以下流程解决这一问题：

```text
发现能力 → 读取规则 → 构建未签名交易 → 模拟并验证
```

它不会替用户签名，也不会直接发送交易，而是负责在签名前构建和检查交易。

未来，Moss 可以应用于：

* 自然语言 DeFi 助手
* 自动化资产管理
* 多步骤链上工作流
* 带 AI 操作入口的 Web3 应用
* Monad 生态中的智能交易基础设施

总体来看，Moss 的价值不只是“让 AI 能调用区块链”，而是让 AI Agent 可以通过一种更标准、更安全、更容易验证的方式与链上协议进行交互。
